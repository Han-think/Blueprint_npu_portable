HOW TO STUDY THIS PACKAGE · using small local LLMs (Ollama + IPEX-LLM on Intel Arc)

1 · 학습 철학

이 패키지는 단순 spec 모음이 아니라 20개 문서가 서로를 참조하는 시스템입니다. 큰 모델 1번에 통째로 던지면 비용도 비싸고 깊이도 떨어집니다. 소형 LLM (7B~14B)을 로컬로 쓰는 본인 환경에 맞춰, 문서를 작게 쪼개 한 번에 한 주제씩 대화하는 방식이 가장 효율적입니다.

CORE LOOP

한 페이지 → 1개 질문 → 모델 답변 → 다른 페이지로 cross-check → 모순 발견 → 학습 노트. 이 사이클이 한 바퀴 = 30분 이내. 하루 2~3 사이클이면 1주일에 패키지 절반 마스터.

2 · 환경 (Ollama + IPEX-LLM)

왜 이 조합인가

  • Ollama: 모델 관리/서빙 표준. ollama pull 한 줄로 모델 교체
  • IPEX-LLM: Intel Arc GPU(또는 Xeon CPU)에서 INT4 양자화로 7B~14B 모델 가속
  • NVIDIA 의존성 없이 본인 워크스테이션에서 로컬 추론 → 비용 0, 프라이버시 ↑
  • 토큰 속도는 cloud보다 느리지만 — 학습 페이스랑 잘 맞음

설치 한 줄 요약

# IPEX-LLM 백엔드의 Ollama 빌드 사용 (Intel GPU 가속)
# https://github.com/intel-analytics/ipex-llm 참고
ollama serve &
ollama pull qwen2.5:14b-instruct-q4_K_M    # 본인 환경 최적
ollama pull qwen2.5-coder:7b-instruct      # 코드 질문용
ollama pull nomic-embed-text               # 임베딩 (RAG용)

3 · 4-Tier 학습 커리큘럼

TIER 0
반나절
지형 파악 — README + Dashboard + Method 만 정독.
질문: "이 패키지의 6개 cluster가 각각 무슨 역할이고, 어떤 순서로 읽어야 하나?"
TIER 1
1~2일
STRUCTURE 클러스터 — Assembly · Joints · Mass. "무엇으로 이루어졌나"를 머릿속에 박는 단계.
질문 예: "P08 throat이 GRCop-42인 이유는? IN718과 뭐가 다른가?" "Hi-Lok과 V-band를 같은 인터페이스에 쓰면 안 되는 이유는?"
TIER 2
3~5일
MATERIAL + ANALYSIS — 제조·검증을 묶어서. 이 단계가 가장 깊이 있는 곳. LPBF/HIP/CMM, MoS 계산, FMEA RPN.
질문 예: "FMEA F-01 (witness line) 의 RPN 160이 어떻게 나왔나? S/O/D 각각 근거는?" "Loads의 case-3 chamber pressure가 Thermal의 hot-side 894K와 동일 시점인가?"
TIER 3
1주
OPS + PROGRAM — Lineage · ECN · ICD · Schedule · Risk. "한 SN이 어떻게 태어나서 죽는가" 의 시간 차원.
질문 예: "ECN-014가 닫혀야 build lock 가능한 이유는? 어느 페이지 어느 항목과 연결되나?" "Risk R-02 (witness line) 가 FMEA F-01과 동일한가? 왜 둘 다 추적하나?"

4 · 질문 템플릿

소형 LLM은 컨텍스트가 짧으므로 1페이지 단위로 붙여넣고 묻습니다.

A. 정의·요약 (Tier 0~1)

다음은 NPU 설계 패키지의 <PageName> 페이지다.

<페이지 본문 붙여넣기>

질문:
1) 이 페이지가 다루는 핵심 개념 3가지를 한 줄씩 정의해라.
2) 항공우주 엔지니어가 처음 보면 헷갈릴 만한 용어 5개와 그 의미.
3) 이 페이지에서 가장 중요한 숫자 1개와 그 의미.

B. Cross-page 검증 (Tier 2)

페이지 A: <Loads.html 본문>
페이지 B: <Thermal.html 본문>

질문:
- A의 load case-3과 B의 hot-side 894K는 동일 운영 시점인가?
- 두 페이지에서 P08 throat을 가리키는 수치가 서로 일관적인가?
- 모순이 있다면 어느 페이지를 신뢰해야 하나?

C. Re-derive (Tier 3)

FMEA의 F-01 (GRCop witness line):
S=8, O=4, D=5, RPN=160

질문:
- S/O/D 각 점수의 정성적 근거를 1~10 스케일로 다시 도출해라.
- 만약 CT 검사를 100% 도입하면 D가 얼마로 떨어지나? RPN은?
- 그 결과 R-02 (Risk page)의 L·I 점수에는 어떤 변동이 생기나?

5 · RAG 미니 셋업 (선택)

20개 문서가 늘어나면 매번 붙여넣기가 귀찮아집니다. 간단한 RAG로 자동화: 

# 페이지를 임베딩 → chroma/lancedb에 저장
ollama embeddings -m nomic-embed-text "<page text>"

# 질문 → top-K 페이지 retrieve → qwen2.5:14b에 컨텍스트 주입
# 직접 작성하기 어렵다면 LangChain/LlamaIndex 의 Ollama 어댑터 사용

청크 사이즈는 800~1200 token, overlap 100. 페이지 단위 자체가 chunk와 잘 맞음.

6 · 학습 노트 포맷

매 사이클마다 짧은 노트 1개. 본인의 학습 데이터셋이 됨:

## <Date> · <PageName> · <1줄 주제>

Q. (LLM에 던진 질문)
A. (모델 답 요약, 2~3줄)

확인 (cross-check):
- <다른 페이지명>의 <섹션>에서 동일/상충 정보 발견
- 결론: 신뢰함 / 모순 있음 / 추가 조사 필요

다음 질문:
- ...

7 · 발전 로드맵

  1. 현재: 1페이지씩 붙여넣기 + Ollama에 질문
  2. 2단계: nomic-embed로 RAG, top-3 페이지 자동 retrieve
  3. 3단계: 본인 학습 노트(Step 6)를 fine-tune 데이터로 → 본인용 도메인 어시스턴트
  4. 4단계: ECN/Risk 자동 분류기 — 새 변경 요청을 어느 페이지에 반영해야 하는지 추천

8 · 흔한 함정

  • 큰 모델을 다 던지지 마라 — 7B로도 페이지 단위면 충분. 모순 발견 위주로.
  • 모델 답을 그대로 믿지 마라 — 항공우주 fastener 같은 곳에서 hallucinate 자주. 반드시 패키지의 다른 페이지에서 cross-check.
  • 한 번에 클러스터 전체를 학습하려 하지 마라 — Tier 단위로 분리. 머리가 식어야 다음 cluster.
  • RAG 너무 일찍 도입하지 마라 — Tier 0~1은 그냥 페이지 정독이 빠름.
